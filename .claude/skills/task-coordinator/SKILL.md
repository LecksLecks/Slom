Разбей сложный запрос на атомарные подзадачи и сформулируй по каждой чёткий
бриф, чтобы вызываемый агент понимал ЧТО делать, КАК и в каких ГРАНИЦАХ — без
уточнений и ошибок.

## Шаг 1: Декомпозиция

Разбей задачу так, чтобы каждая подзадача:
- имела одну конкретную цель;
- была выполнима одним агентом (сверься с ростером в скилле `agent-router`);
- имела чёткие критерии успеха.

## Шаг 2: Контекст для каждой подзадачи

- **Зачем**: почему задача важна в контексте общей цели.
- **Что на входе**: какие данные/файлы/контекст передаются агенту.
- **Что на выходе**: конкретный формат результата (код, отчёт, сигнал).

## Шаг 3: Ограничения

- что агенту **НЕЛЬЗЯ** делать;
- какие источники/данные использовать в приоритете;
- какие крайние случаи предусмотреть;
- дедлайны/лимиты, если применимо.

## Шаг 4: Карточка

```
┌─────────────────────────────────────────────────────────────┐
│ ЗАДАЧА ДЛЯ: [название агента]                                │
├─────────────────────────────────────────────────────────────┤
│ КОНТЕКСТ:                                                    │
│ [2-3 предложения: почему задача существует, что уже сделано, │
│  где результат будет использован]                           │
│                                                             │
│ ЦЕЛЬ:                                                        │
│ [Одно предложение: какой конкретный результат нужен]         │
│                                                             │
│ ВХОДНЫЕ ДАННЫЕ:                                              │
│ - [файл/данные/контекст 1]                                  │
│ - [файл/данные/контекст 2]                                  │
│                                                             │
│ ТРЕБОВАНИЯ:                                                  │
│ ✓ [требование 1]                                            │
│ ✓ [требование 2]                                            │
│ ✓ [требование 3]                                            │
│                                                             │
│ ОГРАНИЧЕНИЯ:                                                 │
│ ✗ [что нельзя делать 1]                                     │
│ ✗ [что нельзя делать 2]                                     │
│                                                             │
│ ФОРМАТ ВЫВОДА:                                               │
│ [Конкретный формат: файл, JSON, отчёт, код с комментариями]  │
│                                                             │
│ КРИТЕРИИ УСПЕХА:                                             │
│ [Как поймём, что задача выполнена правильно]                 │
│                                                             │
│ ЗАВИСИМОСТИ:                                                 │
│ [Что должно быть сделано до/после, связи с другими задачами] │
└─────────────────────────────────────────────────────────────┘
```

## Принципы формулировки

✅ Делай: конкретные измеримые цели («проверь 5 индикаторов», не «проанализируй
рынок»); явные ожидания по формату («Python-файл с функцией X», не «сделай
код»); контекст использования; приоритеты источников; примеры результата, если
возможно.

❌ Не делай: размытых формулировок («сделай хорошо»); неявных ожиданий («как
обычно»); смешения ответственности («найди и закодь» — это две задачи);
игнорирования зависимостей.

## Проверка качества перед выдачей карточки

1. **Понятность** — агент поймёт без уточнений?
2. **Полнота** — все входные данные/контекст есть?
3. **Непротиворечивость** — нет конфликтующих требований?
4. **Выполнимость** — агент может это своими инструментами?
5. **Измеримость** — можно однозначно определить «сделано»?

Хотя бы один ❌ — доработай формулировку, не выдавай карточку.

## Неясные запросы

Если запрос размытый — **не** формулируй карточки. Спроси у пользователя:
какую конечную цель хочет достичь? где/как будет использован результат? есть
ограничения по времени/ресурсам? какие данные/файлы уже есть? что уже
пробовал?

## Пример

«Хочу индикатор пробоя уровней на криптофьючерсах для TradingView.» →

```
┌─────────────────────────────────────────────────────────────┐
│ ЗАДАЧА ДЛЯ: trading-strategy-developer                       │
├─────────────────────────────────────────────────────────────┤
│ КОНТЕКСТ: Пользователь разрабатывает стратегию пробоя         │
│ уровней на криптофьючерсах. Нужен индикатор для TradingView,  │
│ сигнализирующий о пробое ключевых уровней.                   │
│                                                             │
│ ЦЕЛЬ: Pine Script индикатор, отмечающий пробой дневного       │
│ high/low с подтверждением объёмом.                           │
│                                                             │
│ ВХОДНЫЕ ДАННЫЕ: timeframe = 15m, lookback = 20                │
│                                                             │
│ ТРЕБОВАНИЯ:                                                  │
│ ✓ Линии дневного high/low                                   │
│ ✓ Сигнал при пробое с закрытием свечи за уровнем            │
│ ✓ Фильтр: объём свечи > 1.5 × средний объём(20)             │
│                                                             │
│ ОГРАНИЧЕНИЯ: ✗ Без repainting  ✗ Без торгового функционала    │
│ ФОРМАТ ВЫВОДА: .pine-файл + краткие комментарии               │
│ КРИТЕРИИ УСПЕХА: вставляется в TradingView без ошибок         │
│ ЗАВИСИМОСТИ: после выполнения → work-verifier                │
└─────────────────────────────────────────────────────────────┘
```

## Ответ пользователю

Покажи план в этой структуре, затем вызывай агентов по порядку через `Agent`:

**📋 Обзор задачи** — краткое описание общей цели.
**🎯 Декомпозиция** — 1. [подзадача] → [агент], 2. ..., 3. ...
**📦 Задачи для агентов** — карточка(и) из Шага 4.
**⚠️ Заметки координатора** — зависимости/риски, что проверить после.

Экономь токены: без вводных фраз, таблицы вместо прозы — см. «Экономия
токенов» в `CLAUDE.md`.
